Архитектура, принцип работы и безопасная эксплуатация

Подробное руководство для инженера: что именно сканирует и удаляет приложение, зачем нужен каждый источник, как устроена модель безопасности и где заканчивается гарантированный результат очистки.

Назначение: поиск и удаление следов USB/Type‑C устройств и связанных forensic‑артефактов на работающей Windows; дополнительно — аудит сети и универсальная очистка журналов событий.

Состояние проекта: 18 июля 2026 года; версия приложения 1.7.1; руководство сверено с текущей архитектурой исходного кода.

Уровень: Senior Engineer / DFIR-aware system engineer.

Принцип: очистка — деструктивная операция. Приложение показывает, что нашло и что удаляет; оно не восстанавливает данные автоматически и не гарантирует, что Windows или сторонняя защита не воссоздадут часть следов после перезагрузки.

Как пользоваться этим руководством

Документ отвечает не на вопрос «в каком классе написан метод», а на практические вопросы инженера: что ищется, почему это удаляется, в каком порядке идёт очистка, какие гарантии даёт резервная копия и что нельзя утверждать после успешного «0 оставшихся USBSTOR».

Главное правило эксплуатации. USBTraceCleaner — средство удаления артефактов, а не forensic‑аудит. Успешная очистка означает, что выбранные и доступные следы были удалены или перезаписаны в рамках реализованных шагов. Она не доказывает, что устройство «никогда не подключалось», и не отменяет необходимость образа диска, если исследование юридически значимо.

1. Назначение и границы приложения

USBTraceCleaner — локальная программа с графическим интерфейсом (GUI) для Windows 10/11. Она находит сохранившиеся следы USB‑накопителей и связанных forensic‑артефактов, позволяет выбрать элементы и удалить их с резервной копией реестра. Отдельные вкладки закрывают не‑storage USB‑следы, сетевые артефакты и универсальную работу с журналами Event Viewer.

Приложение позиционируется как расширенный аналог USB Oblivion: ядро очистки реестра PnP/USBSTOR дополнено Prefetch, Amcache, Shimcache, Jump Lists, Recycle Bin, VSS, System log и удалением self‑trace утилит очистки.

Какие задачи приложение помогает закрыть

  • Какие следы USB‑накопителей Windows хранит в реестре, логах и пользовательских профилях?
  • Каковы VID/PID, производитель, модель и дата первого подключения по PnP‑свойству?
  • Какие «призраки» PnP (дубликаты и orphan‑экземпляры) остались после отключения устройств?
  • Можно ли выборочно удалить найденные артефакты с манифестом и .reg‑бэкапом?
  • Как очистить не‑storage USB‑следы (хабы, камеры, прочие VID/PID в usbflags/DeviceMigration)?
  • Какие сетевые профили Wi‑Fi/VPN/NLA и связанные журналы присутствуют на машине?
  • Как просмотреть и очистить произвольные каналы Event Log, включая System?
  • Как выполнить полный forensic‑путь без GUI через --clean?

Чем приложение не является

  • Это не write blocker, не средство создания forensic image и не замена EnCase, X‑Ways, Magnet AXIOM или Autopsy.
  • Это не UsbForensicAudit: приложение не строит граф идентичности устройств, не считает EvidenceStrength/Coverage и не предназначено для доказательной интерпретации истории подключений.
  • Оно не гарантирует полное исчезновение всех возможных следов USB во всех теневых копиях, offline‑образах, бэкапах стороннего ПО и корпоративных SIEM.
  • Оно не удаляет встроенные USB‑хабы ROOT_*, onboard HID/UVC/BT как «пользовательские флешки» — они намеренно защищены.
  • Оно не поддерживает Windows, установленную на USB‑диске, и не рассчитано на очистку при подключённых съёмных накопителях.
  • Автоматического undo нет: восстановление — только ручной импорт .reg и штатные средства Windows.
Основа методики очистки: логика USB Oblivion / USBDeview‑совместимые пути реестра · Magnet Forensics — USB Devices Artifact Profile · Microsoft — SetupAPI Text Logs · MITRE ATT&CK — Clear Windows Event Logs

3. Полный конвейер сканирования и очистки

3.1. Конвейер сканирования (вкладка USB)

Сканирование идёт последовательно. Порядок важен: сначала инвентаризация ControlSet и статических ключей, затем пользовательские профили и файловые артефакты, в конце — ghosts PnP.

ШагЧто происходитЗачем
1Обход всех ControlSet00x: DeviceContainers, Enum\USB, Enum\USBSTOR, DeviceClasses, STORAGE\Volume, WPDBUSENUM, static keys, DeviceMigration.Собрать основной инвентарь PnP/USB storage.
2HKLM static: Setup\Upgrade, EMDMgmt, VolumeInfoCache, MuiCache, WBEM/WDM values с USBSTOR.Поймать остаточные ссылки вне Enum.
3MountedDevices — USB/WPD values и orphan‑буквы D–Z.Связать тома и буквы со съёмными устройствами.
4Профили HKU: MountPoints2, Autoplay, Portable Devices, ShellBags, UserAssist (фильтр).Удалить пользовательский контекст томов.
5BAM/DAM — буквы D–Z на отключённых дисках + self‑trace.Убрать execution‑следы съёмных путей и утилит очистки.
6Файлы логов: setupapi*, INFCACHE.1, wmiprov.log, PCA.Убрать текстовые следы установки устройств.
7Фиксированный список PnP/storage Event Log каналов.Подготовить очистку журналов, где пишутся USB‑события.
8Recent LNK и Jump Lists с фильтрацией по содержимому.Удалить shell‑ссылки на съёмные тома.
9Extended forensic: Prefetch, Amcache, Shimcache, Explorer MRU, Recycle Bin.Закрыть типичные forensic‑источники после USB Oblivion‑ядра.
10PnPGhostScanner (если включён ScanPnPGhosts).Найти дубликаты и orphan‑экземпляры Enum\USB/USBSTOR.

Дедупликация ключом Type|Location|ValueName. По умолчанию все найденные элементы получают Selected = true.

3.2. Конвейер очистки (вкладка USB)

Предусловия: права администратора и Windows 10/11. Дальше шаги выполняются в фиксированном порядке.

ШагЧто происходитЗачем / риск
1Опциональная точка восстановления (Checkpoint-Computer). В GUI по умолчанию выкл.Restore point сам оставляет forensic‑артефакт.
2Резервная копия .reg + manifest .txt рядом с EXE (по умолчанию вкл.).Даёт ручной откат реестровой части.
3Завершение Explorer (если не запланирован reboot).Снизить блокировки ShellBags/Recent.
4Остановка служб: vds, SysMain, EMDMgmt, WMPNetworkSvc.Освободить ReadyBoost/тома.
5Core USB cleanup: offline USB disks → uninstall USBSTOR → UsboOblivionCleanup по ControlSet → MountedDevices.Удалить «живой» PnP‑инвентарь накопителей.
6Выбранные артефакты: values → keys (deepest first) → files.Дочистить то, что ядро не покрыло.
7Event logs: wevtutil cl + ExtraChannels (UserPnp / Kernel‑PnP / System purge).Убрать события подключения/установки.
8Offline hive retry для USBSTOR.Дожать ключи, занятые живой системой.
9VSS: vssadmin delete shadows /all /quiet (если включено).Уничтожает все теневые копии.
10SelfTraceCleaner — следы USBTraceCleaner, USBOblivion, USBDeview, USBDetector, UsbForensicAudit.Убрать доказательства запуска очистителей.
11Verify: CountUsbStorageTraceDevices().Критерий успеха: 0 оставшихся USBSTOR‑следов (или simulation).
12PnP ghosts из выбора / RemoveAll.Удалить дубликаты и orphan‑экземпляры.
13Restore setupapi.dev.log, если файл уничтожен scrub‑ом.Не оставлять систему без критичного лога установки.
14Purge System.evtx при CleanSystemEventLog — убрать Event ID 104 от очистки журналов.Деструктивно для всего System log.
15Reboot (shutdown /r /t 5) или перезапуск Explorer.Зафиксировать состояние после удаления PnP.
Режим симуляции. Галочка «Симуляция (без удаления)» на вкладке USB включает CleanupOptions.SimulationMode: выполняются те же шаги с логом [SIM], но реестр, файлы и журналы не изменяются. На вкладке «Аудит сети» симуляция включается отдельной галочкой.

4. Функциональные блоки: что, зачем и как выполняется

4.1. Права администратора и совместимость ОС

Что проверяется. Манифест requireAdministrator; runtime‑проверка AdminHelper.IsAdministrator() и IsWindows10Or11() (build ≥ 10240; для Windows 11 — ≥ 22000).

Зачем. Без повышения прав недоступны Security/часть Event Log, удаление защищённых ключей Enum, offline hive, purge System.evtx и остановка служб.

Что означает результат. Статус администратора — необходимое, но не достаточное условие: политики, EDR и занятые устройства всё равно могут дать Access Denied.

Источники: Microsoft — UAC

4.2. Ядро реестра: USBSTOR, USB, DeviceClasses, usbflags

Что сканируется/удаляется. Во всех ControlSet:

  • Enum\USBSTOR, Enum\USBPRINT, Services\USBSTOR\Enum — полное удаление;
  • Enum\SWD\WPDBUSENUM, Enum\STORAGE\Volume, RemovableMedia — фильтр по USBSTOR/съёмным;
  • Enum\USB\VID_*&PID_* — экземпляры с Service USBSTOR / UASPStor / MTP / HID по опциям;
  • Control\DeviceContainers — только если BaseContainers содержит USBSTOR;
  • Control\DeviceClasses и известные GUID томов/WPD — USBSTOR‑filtered;
  • Control\usbflags, IgnoreHWSerNum*, Control\usbstor;
  • Control\DeviceMigration — storage/MTP записи, не весь корень;
  • ReadyBoost (Services\rdyboost).

Зачем. Это основной инвентарь, по которому Windows «помнит» флешки и MTP‑тома.

Защита. ROOT_* хабы не удаляются — полное удаление вызывает повторное перечисление PnP и риск нестабильности шины. DeviceContainers не чистятся по голому USB#, чтобы не задеть камеру/клавиатуру.

Источники: Microsoft — Device identification strings · NirSoft USBDeview

4.3. MountedDevices, WPD и пользовательские профили

Что затрагивается.

  • HKLM\SYSTEM\MountedDevices — USB/WPD mappings и orphan‑буквы;
  • SOFTWARE\Microsoft\Windows Portable Devices\Devices\{child} — только дочерние USB/WPD, родительский ключ сохраняется;
  • HKU: MountPoints2, CPC\Volume, Autoplay KnownDevices, Portable Devices, ShellBags, MuiCache, SyncMgr;
  • UserAssist — только значения с USB/self‑маркерами после ROT13;
  • Explorer MRU: RecentDocs, TypedPaths, OpenSavePidlMRU, LastVisitedPidlMRU.

Как понимать. Удаление MountPoints2/LNK убирает пользовательский след тома, но не доказывает и не опровергает копирование файлов. Одна буква диска никогда не была уникальным идентификатором устройства.

Источники: Magnet Forensics — MountedDevices / USBSTOR

4.4. SetupAPI, INFCACHE, PCA и scrub логов

Что делается. Файлы %WinDir%\setup*.log, %WinDir%\inf\setupapi*.log, setupapi.ev1/2/3, INFCACHE.1, wmiprov.log, PCA PcaGeneralDb*.txt сканируются и scrub/delete по правилам.

Важная оговорка. setupapi.dev.log намеренно не уничтожается «в ноль» навсегда: после scrub приложение восстанавливает критичный файл через LogFileScrubber.EnsureCriticalSetupApiLogs, чтобы Windows Setup/PnP не остались без журнала. CLI --restore-logs пересоздаёт пустой setupapi.dev.log.

Источники: Microsoft — SetupAPI Text Logs

4.5. Prefetch, Amcache, Shimcache, Jump Lists, Recycle Bin

Что сканируется. Через ExtendedForensicScanner и ForensicTracePatterns:

  • Prefetch *.pf с USB/self‑токенами;
  • Amcache.hve — selective scrub через копию и offline load (AmcacheCleaner);
  • AppCompatCache / Shimcache — value переписывается, если содержит USB/self;
  • Recent *.lnk, AutomaticDestinations / CustomDestinations — только при совпадении содержимого;
  • $Recycle.Bin\$I*/$R* с USB‑путями.

Ограничение. Amcache может быть залочен; без успешного offline‑load scrub пропускается. Jump List без USB/self‑маркера не трогается.

Источники: Eric Zimmerman Tools · Velociraptor — BAM

4.6. BAM/DAM и self‑trace

Что удаляется. В Services\bam|dam\State\UserSettings\<SID> — values вида X:\... для букв D–Z на отключённых дисках и пути известных cleaner‑утилит.

SelfTraceCleaner дополнительно чистит Prefetch/BAM/Recent/Jump Lists/Temp для:

  • USBTraceCleaner;
  • USB Oblivion;
  • USBDeview;
  • USBDetector;
  • UsbForensicAudit.

Как понимать. Это снижает видимость факта очистки на самой машине. Это не скрывает запись в централизованном SIEM/EDR и не отменяет необходимость внешней chain of custody.

4.7. Журналы событий (вкладка USB)

Фиксированный список каналов для USB‑вкладки:

  • Microsoft-Windows-DeviceSetupManager/Operational, Admin
  • Microsoft-Windows-Kernel-PnP/Configuration
  • Microsoft-Windows-Kernel-ShimEngine/Operational
  • Microsoft-Windows-DriverFrameworks-UserMode/Operational
  • Microsoft-Windows-Partition/Diagnostic
  • Microsoft-Windows-Storage-ClassPnP/Operational
  • Microsoft-Windows-WPD-MTPClassDriver/Operational
  • Microsoft-Windows-UserPnp/DeviceInstall, Operational
  • HardwareEvents

Дополнительно при clean: UserPnp/Operational, Kernel‑PnP Device Management / Device Configuration; при опции — полный purge System.evtx (чтобы убрать Event ID 104 от очистки журналов).

Механизм: wevtutil cl с таймаутом ~12 с; для System — остановка службы EventLog, удаление файла или MoveFileEx до reboot.

Риск. Очистка System удаляет и несвязанные с USB события. Повторный wevtutil cl снова может породить Event ID 104 — именно поэтому есть отдельный purge файла журнала.

4.8. PnP ghosts: дубликаты и orphan‑экземпляры

Что ищется. PnPGhostScanner:

  • Duplicates — одинаковый ControlSet|VidKey|DuplicateGroupKey; keeper = live device или наибольший enum index;
  • Orphans — removable USB service + устройство отсутствует по SetupAPI/presence check;
  • отдельные orphan в Enum\USBSTOR.

CLI: --fix-duplicates запускает ghost‑cleaner без полного forensic‑пути.

Как понимать. «Призрак» — остаточная PnP‑запись, а не доказательство повторного подключения в конкретную секунду.

4.9. Вкладка «Другие USB-следы»

Назначение. Артефакты вне storage‑сканера — по аналогии с блоком «Другие следы» в USBDetector: не‑накопители в usbflags / DeviceMigration / Setup\Upgrade.

Источники: Control\usbflags (compact VIDPID + IgnoreHWSerNum), Control\DeviceMigration, SYSTEM\Setup\Upgrade\...\DeviceMigration.

Исключение: всё, что матчит StorageTracePatterns (USBSTOR/UASPStor и т.п.), во вкладку не попадает — это зона основной USB‑вкладки.

Очистка: сбор путей по VID/PID → uninstall Enum\USB instances → delete registry (ownership + native + SYSTEM retry) → scrub строк setupapi с этим VID/PID.

Побочный эффект. USB‑мышь/клавиатура могут кратковременно отключиться. «Найдено: 0» после частичной очистки не гарантирует полный wipe всех упоминаний VID/PID.

4.10. Вкладка «Журналы событий»

Независимый модуль — не меняет логику ArtifactCleaner.

  • Список: динамически все каналы EventLogSession.GetLogNames();
  • группы: «Журналы Windows», Microsoft, vendor‑prefix, «Прочие»;
  • чтение до 400 записей, опциональный USB‑фильтр;
  • очистка: wevtutil cl; Security — только с галочкой; System purge — отдельная опция.

Классические каналы Windows: Application, Security, Setup, System, ForwardedEvents.

4.11. Device uninstall, offline disks и ACL bypass

Что выполняется перед удалением ключей.

  • VolumeHelper.OfflineUsbDisks() — PowerShell Set-Disk -IsOffline;
  • DeviceUninstallHelper — SetupAPI CM_Uninstall_DevNode / pnputil /remove-device;
  • при Access Denied — RegistrySecurityHelper.TakeOwnershipTree и удаление от SYSTEM через RegistrySystemHelper (до 3 попыток);
  • offline SYSTEM hive copy в %TEMP%\USBTC_SYSTEM_{guid} для retry USBSTOR.

Зачем. Живые PnP‑узлы и TrustedInstaller‑ACL часто блокируют обычный RegDeleteKey.

4.12. Volume Shadow Copies

Что делается. При включённой опции — vssadmin delete shadows /all /quiet.

Зачем. Иначе старые USB‑записи могут остаться в shadow copy SYSTEM/SOFTWARE hive или SetupAPI.

Последствие. Теряются все теневые копии тома, включая точки восстановления, не связанные с USB. Включайте опцию только осознанно.

4.13. База VID/PID и отображение в UI

Что используется. Embedded ресурс Assets/USBVendors.txt (~3400 VID, десятки тысяч продуктов) через UsbVendorDatabase.

Что показывается. Колонки VID, PID, производитель, модель, первое подключение (PnP property {83da6326-...}\0064 / FirstInstallDate), источник.

Как понимать. VID/PID описывает модель/семейство, а не уникальный физический экземпляр. Для экземпляра нужен serial / instance ID.

Источники: The USB ID Repository · DEVPKEY_Device_FirstInstallDate

5. Как выполняется отбор и классификация

5.1. Признак «USB‑накопитель»

ArtifactClassifier.IsUsbStorageTrace считает запись storage‑следом, если:

  • путь содержит \Enum\USBSTOR, \Services\USBSTOR\, \Control\usbflags\, \Control\usbstor\, \Control\DeviceMigration, \Enum\USBPRINT;
  • текст содержит USBSTOR / UASPStor;
  • тип RegistryMounted;
  • \Enum\USB\ (не ROOT_) и Service = USBSTOR/UASPStor.

Родительский ключ WPD Devices в storage‑группу не попадает.

5.2. Forensic content matching

ForensicTracePatterns ищет токены: USBSTOR, WPDBUSENUM, UASPStor, RemovableMedia, WpdMtp, имена cleaner‑exe, съёмные буквы D–Z (если диск сейчас не подключён).

5.3. Модель выбора

  • Каждая строка имеет checkbox Selected (по умолчанию true).
  • Sidebar фильтрует по ArtifactViewGroup.
  • Clean обрабатывает только выбранные элементы + всегда выполняет core USB registry path.
  • IsHandledByCoreUsbCleanup() пропускает дубли, уже закрытые ядром.

5.4. Разделение USB vs «Другие USB-следы»

ВкладкаВключаетИсключает
USBUSBSTOR/UASP/WPD‑тома, MountedDevices, forensic files, PnP ghosts storageНе‑storage usbflags как отдельный продуктовый список
Другие USB-следыusbflags / DeviceMigration / Setup\Upgrade без storage‑маркеровВсё, что матчит StorageTracePatterns
Практический смысл. Сначала чистите вкладку USB (накопители), затем при необходимости «Другие USB-следы». Смешение групп в одном проходе не требуется: это разные цели и разный риск для HID/камер.

6.1. Как читать результат очистки

  • Success + 0 USBSTOR — ядро storage‑следов в доступных ControlSet снято по критерию счётчика.
  • Частичные ошибки Access Denied — устройство занято, ACL или политика; смотрите журнал и повторите после reboot / offline hive.
  • «Найдено: 0» на «Других USB» — не гарантия полного wipe конкретного VID/PID во всех логах.
  • Event ID 104 после очистки журналов — ожидаемый артефакт очистки; снимается purge System.evtx, но может вернуться при новых clear.
Перед юридически значимым исследованием сначала снимите forensic‑образ штатным процессом организации. Запуск USBTraceCleaner на живой системе сам создаёт и удаляет артефакты и меняет состояние доказательственной базы.

7. Аудит сети

Отдельная вкладка «Аудит сети» не является частью USB‑конвейера. Она инвентаризирует и при необходимости удаляет сетевые следы Windows.

7.2. Full clean сети

NetworkPostCleanActions может: остановить DPS/DiagTrack, отключить VPN, удалить все Wi‑Fi профили и XML, очистить деревья NetworkList, arp -d *, ipconfig /flushdns, отключить Wi‑Fi session, запланировать reboot через 60 с и выполнить NetworkAuditVerifier.

Белый список IP / SSID / VPN маркирует элементы как «Разрешено» и исключает их из очистки (выборочной и максимальной), включая удаление Wi‑Fi профилей в глубоком пути. Сброс hosts ломает локальный DNS Docker/Kubernetes, если там были кастомные записи.

7.3. Отчёты сети

PDF через QuestPDF: ReportExportService.BuildNetworkReport + summary‑окно с хронологией подключений.

Источники: Microsoft — Network Location Awareness · Microsoft — netsh wlan

8. Практический порядок работы инженера

8.1. Перед запуском

  1. Зафиксировать цель: удаление следов конкретных накопителей, подготовка стенда, сетевой аудит или очистка журналов.
  2. Отключить все USB‑накопители. Не запускать на Windows‑on‑USB.
  3. Записать локальное время, часовой пояс и состояние ПК.
  4. Если нужен forensic‑аудит истории — сначала UsbForensicAudit / образ диска, и только потом cleaner.
  5. Запустить EXE от имени администратора.

8.2. Типовой сценарий очистки USB

  1. Вкладка USB → «Сканировать».
  2. Просмотреть группы sidebar; снять галочки с того, что нельзя трогать.
  3. Проверить опции: backup on; restore point обычно off; VSS и System log — только при полном forensic‑пути.
  4. «Очистить» → подтвердить → дождаться verify.
  5. Сохранить PDF‑отчёт при необходимости.
  6. Перезагрузить Windows (обязательно после реальной очистки).
  7. После reboot при необходимости повторить скан; для setupapi — --restore-logs.

8.3. Сценарий «Другие USB-следы»

  1. Сначала завершить storage‑очистку.
  2. Сканировать вкладку «Другие USB-следы».
  3. Выбрать VID/PID группы; учесть риск для мыши/клавиатуры.
  4. Очистить и перезагрузить.

8.4. Сценарий сети

  1. Сканировать → изучить Wi‑Fi/VPN/NLA/hosts/SRU.
  2. Экспортировать PDF summary до очистки, если нужен отчёт «как было».
  3. Выборочная очистка или full clean с пониманием disconnect Wi‑Fi и reboot 60 с.
  4. Проверить verifier / повторный скан.

8.5. Что писать в инженерном заключении

Корректно: «После выполнения USBTraceCleaner v1.7.1 на хосте X счётчик USBSTOR‑следов стал 0. Созданы backup/manifest рядом с EXE. Выполнена перезагрузка. Повторный скан не показал Enum\USBSTOR записей съёмных накопителей».

Некорректно: «Устройство никогда не подключалось» или «все возможные следы USB уничтожены во всех системах резервного копирования».

Факторы, снижающие полноту очистки

  • USB‑накопитель оставался подключённым во время clean;
  • ключ/файл залочен службой или EDR;
  • Amcache/SRUDB требуют reboot‑retry;
  • теневые копии не удалялись (опция VSS выкл.);
  • профиль пользователя недоступен / зашифрован;
  • корпоративный агент пишет свои журналы вне Windows Event Log;
  • повторная установка драйвера/подключение устройства создаёт новые следы.

Итог для инженера

USBTraceCleaner следует рассматривать как управляемое средство удаления USB/forensic‑артефактов на живой Windows. Его сильная сторона — не «магическая кнопка стереть всё», а явный конвейер: скан → классификация → бэкап/manifest → core USBSTOR cleanup → выбранные артефакты → журналы/VSS/self‑trace → verify → reboot.

Надёжная эксплуатация получается, когда инженер заранее отключает накопители, понимает деструктивность VSS/System log, сохраняет backup/manifest, перезагружает систему и не подменяет результат очистки утверждениями о полной исторической невозможности подключения устройства.
